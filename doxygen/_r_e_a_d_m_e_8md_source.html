4 ## Usage
5 The basic use of libasynql has 5 steps:
6 1. Add default database settings in your `config.yml`.
7 2. Write down all the SQL queries you will use in a resource file
8 3. Initialize the database in `onEnable()`.
9 4. Finalize the database in `onDisable()`.
10 5. Obviously, and most importantly, use libasynql in your code.
11 
12 ### Configuration
13 To let the user choose what database to use, copy the following into your default `config.yml`. Remember to change the default schema name under `mysql`.
14 
15 ```yaml
16 database:
17  # The database type. "sqlite" and "mysql" are supported.
18  type: sqlite
19 
20  # Edit these settings only if you choose "sqlite".
21  sqlite:
22  # The file name of the database in the plugin data folder.
23  # You can also put an absolute path here.
24  file: data.sqlite
25  # Edit these settings only if you choose "mysql".
26  mysql:
27  host: 127.0.0.1
28  # Avoid using the "root" user for security reasons.
29  username: root
30  password: ""
31  schema: your_schema
32 ```

34 ### Initialization and Finalization
35 libasynql simplifies the process of initializing a database into a single function call.
36 
37 ```php
38 use pocketmine\plugin\PluginBase;
39 use poggit\libasynql\libasynql;
40 
41 class Main extends PluginBase{
42  private $database;
43 
44  public function onEnable(){
45  $this->saveDefaultConfig();
46  $this->database = libasynql::create($this, $this->getConfig()->get("database"), [
47  "sqlite" => "sqlite.sql",
48  "mysql" => "mysql.sql"
49  ]);
50  }
51 
52  public function onDisable(){
53  if(isset($this->database)) $this->database->close();
54  }
55 }
56 ```
57 
58 The [`\poggit\libasynql\libasynql::create()`](https://poggit.github.io/libasynql/doxygen/classpoggit_1_1libasynql_1_1libasynql.html#add1449f8fef87cc275a3d99f2440c642) method accepts 3 parameters:
59 - Your plugin main
60 - The config entry where the database settings should be found
61 - An array for your SQL files. For each SQL dialect you are supporting, use it as the key, and use the path (or array of paths, relative to the `resources` folder) of the SQL files as the value. We are going to create them in the next step.
62 
63 It returns a [`\poggit\libasynql\DataConnector`](https://poggit.github.io/libasynql/doxygen/interfacepoggit_1_1libasynql_1_1_data_connector.html) object, which is the main query interface.
64 
65 In case of error, a ConfigException or an SqlError will be thrown. If not caught by the plugin, this will go straight out of onEnable() and disable the plugin. Therefore, make sure to check `isset($this->database)` before calling `$this->database->close()` in onDisable().
66 
67 ### Creating SQL files
68 In the resources file, create one file for each SQL dialect you are supporting, e.g. `resources/sqlite.sql` and `resources/mysql.sql`.
69 
70 Write down all the queries you are going to use in each file, using the [Prepared Statement File format](#prepared-statement-file-format).
71 
72 ### Calling libasynql functions
73 Finally, we are prepared to use libasynql in code!
74 
75 There are 4 query modes you can ues: GENERIC, CHANGE, INSERT and SELECT.
76 - GENERIC: You don't want to know anything about the query except whether it is successful. You may want to use this in `CREATE TABLE` statements.
77 - CHANGE: Your query modifies the database, and you want to know how many rows are changed. Useful in `UPDATE`/`DELETE` statements.
78 - INSERT: Your query is an `INSERT INTO` query for a table with an `AUTO_INCREMENT` key. You will receive the auto-incremented row ID.
79 - SELECT: Your query expects a result set, e.g. a `SELECT` statement, or reflection queries like `EXPLAIN` and `SHOW TABLES`. You will receive a `SqlSelectResult` object that represents the columns and rows returned.
80 
81 They have their respective methods in DataConnector: `executeGeneric`, `executeChange`, `executeInsert`, `executeSelect`. They require the same parameters:
82 
83 - The name of the prepared statement
84 - The variables for the query, in the form of an associative array "variable name (without colon)" => value
85 - An optional callable triggered if the query succeeded, accepting different arguments:
86  - GENERIC: no arguments
87  - CHANGE: `function(int $affectedRows)`
88  - INSERT: `function(int $insertId, int $affectedRows)`
89  - SELECT: `function(SqlSelectResult $result)`
90 - An optional callable triggered if an error occurred. Can accept an `SqlError` object.

92 ## Prepared Statement File Format
93 A Prepared Statement File (PSF) contains the queries that a plugin uses. The content is valid SQL, so it is OK to edit with a normal SQL editor.
94 
95 The PSF is annotated by "command lines", which start with `-- #`, followed by the command symbol, then the arguments. Between the `#` and the command symbol, there can be zero to infinite spaces or tabs; between the command symbol and the arguments, there can also be zero to infinite spaces or tabs. Between every two arguments, one to infinite spaces or tabs are required.
96 
97 ### Dialect declaration
98 A PSF always starts with a dialect declaration.
99 
100 #### Symbol
101 `!`
102 
103 #### Arguments
104 ##### DIALECT
105 Possible values: `mysql`, `sqlite`
106 
107 #### Example
108 
109 ```sql
110 -- #! mysql
111 ```
112 
113 ### Group declaration
114 Queries may be organized by groups. Each group has an identifier name, and a group can be stacked under another. Groups and queries under a group will be prepended the parent group's identifier plus a period in their own identifiers.
115 
116 For example, if a parent group declares an identifier `foo`, and the child group/query declares an identifier `bar`, the real identifier for the child group/query is `foo.bar`.
117 
118 Duplicate group identifier declarations are allowed, as long as the resultant queries do not have identical full identifiers.
119 
120 #### Symbol
121 - Start: `{`
122 - End: `}`
123 
124 #### Arguments (Start)
125 ##### IDENTIFIER_NAME
126 The name of this group.
127 
128 All characters except spaces and tabs are allowed, including periods.
129 
130 #### Example
131 
132 ```sql
133 -- #{ group.name.here
134  -- #{ child.name
135  -- the identifier of the child group is "group.name.here.child.name"
136  -- #}
137 -- #}
138 ```
139 
140 Note that PSF is insensitive about spaces and tabs, so this variant is equivalent:
141 
142 ```sql
143 -- #{ group.name.here
144 -- # { child.name
145  -- the identifier of the child group is still "group.name.here.child.name"
146 -- # }
147 -- #}
148 ```
149 
150 ### Query declaration
151 A query is declared like a group. A query does not need to belong to a group, because the query can declare the periods in its own identifier, which has equivalent effect as groups.
152 
153 Child groups are not allowed in a query declaration. In other words, a `{}` pair either has other group/query declarations inside, or has query text (and optionally variable declarations) inside. It cannot have both.
154 
155 #### Symbol
156 - Start: `{` (same as group declaration)
157 - End: `}`
158 
159 #### Arguments
160 Same arguments as a group declaration.
161 
162 ### Variable declaration
163 A variable declaration declares the required and optional variables for this query. It is only allowed inside a query declaration.
164 
165 #### Symbol
166 - `:`
167 
168 #### Arguments
169 ##### VAR_NAME
170 The name of the variable. Any characters apart from spaces, tabs and colons are allowed. However, to comply with ordinary SQL editors, using "normal" symbols (e.g. variable names in other programming languages) is recommended.
171 
172 ##### VAR_TYPE
173 The variable type. Possible values:
174 - `string`
175 - `int`
176 - `float`
177 - `bool`
178 
179 ##### VAR_DEFAULT
180 If the variable is optional, it declares a default value.
181 
182 This argument is not affected by spaces. It starts from the first non-space non-tab character after VAR_TYPE, and ends before the trailing space/tab characters of the line
183 
184 ###### `string` default
185 There are two modes, literal string and JSON string.
186 
187 If the argument starts with a `"` and ends with a `"`, the whole argument will be parsed in JSON. Otherwise, the whole string is taken literally.
188 
189 ###### `int` default
190 A numeric value that can be parsed by [`(int)` cast, equivalent to `intval`](https://php.net/intval).
191 
192 ###### `float` default
193 A numeric value that can be parsed by [`(float)` cast, equivalent to `floatval`](https://php.net/floatval).
194 
195 ###### `bool` default
196 `true`, `on`, `yes` or `1` will result in true. Other values, as long as there is something, will result default false. (If there is nothing, the variable will not be optional)
197 
198 ### Query text
199 Query text is not a command, but the non-commented part between the start and end commands of a query declaration.
200 
201 Variables are interpolated in query text using the `:var` format. Note that libasynql uses a homebrew algorithm for identifying the variable positions, so they might be inaccurate.
202
